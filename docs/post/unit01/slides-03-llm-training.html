Part 1: Tokenization

What is tokenization?

Models can only process numbers. Tokenization converts sequences into integer IDs.

Level Example input Vocabulary size
Character A C G T a c g t ... ~65 (Shakespeare), ~16 (DNA)
Sub-word (BPE) "the", "##ing", … ~50,000 (GPT-2)
k-mer ACG, CGT, … \(4^k\)

For our nanoGPT: character-level — simple and shows the core ideas.

Character tokenization in practice

chars = sorted(list(set(text)))    # all unique characters
vocab_size = len(chars)            # 65 for Shakespeare

stoi = { ch: i for i, ch in enumerate(chars) }  # char → int
itos = { i: ch for i, ch in enumerate(chars) }  # int → char

encode = lambda s: [stoi[c] for c in s]
decode = lambda l: ''.join([itos[i] for i in l])

# encode entire dataset
data = torch.tensor(encode(text), dtype=torch.long)

Part 2: The Loss Function

Cross-entropy loss

The model outputs logits — one score per vocabulary token. Cross-entropy measures how wrong the prediction is.

\[\mathcal{L} = -\log P(\text{correct token})\]

  • Model assigns probability 0.78 to correct character → loss = \(-\log(0.78) \approx 0.25\) (good)
  • Model assigns probability 0.01 to correct character → loss = \(-\log(0.01) \approx 4.6\) (bad)

What does the loss tell us?

For a random model with vocabulary size \(V\), every token gets probability \(\frac{1}{V}\):

\[\mathcal{L}_{\text{random}} = -\log\left(\frac{1}{V}\right) = \log(V)\]

Vocabulary Random loss
DNA (4 chars) \(\log(4) \approx 1.39\)
DNA + case (10 chars) \(\log(10) \approx 2.30\)
Shakespeare (65 chars) \(\log(65) \approx 4.17\)

This is your baseline. A trained model should beat it. If your loss is above this after training, something is wrong.

From logits to probabilities to loss

def forward(self, idx, targets=None):
    logits = self.lm_head(x)          # (B, T, vocab_size)

    if targets is not None:
        B, T, C = logits.shape
        logits  = logits.view(B*T, C)    # flatten
        targets = targets.view(B*T)      # flatten
        loss = F.cross_entropy(logits, targets)

    return logits, loss

PyTorch’s F.cross_entropy applies softmax internally — pass raw logits, not probabilities.

Part 3: The Training Loop

The standard loop

optimizer = torch.optim.AdamW(model.parameters(), lr=learning_rate)

for step in range(max_iters):
    # 1. sample a batch
    xb, yb = get_batch('train')

    # 2. forward pass
    logits, loss = model(xb, yb)

    # 3. backward pass
    optimizer.zero_grad()
    loss.backward()

    # 4. update weights
    optimizer.step()

Same loop you saw in unit00 — the architecture changes but the training recipe doesn’t.

Monitoring training: estimate_loss

Don’t just watch training loss — check the validation loss periodically:

@torch.no_grad()
def estimate_loss():
    out = {}
    model.eval()
    for split in ['train', 'val']:
        losses = torch.zeros(eval_iters)
        for k in range(eval_iters):
            X, Y = get_batch(split)
            logits, loss = model(X, Y)
            losses[k] = loss.item()
        out[split] = losses.mean()
    model.train()
    return out
  • Train loss ↓, val loss ↓ → model is learning
  • Train loss ↓, val loss ↑ → overfitting

What happens during training?

Starting from random weights, loss starts near \(\log(V)\).

As the model trains:

  1. Bigram statistics are learned first (fast)
  2. Local patterns (common words/motifs)
  3. Long-range dependencies (slow, requires deep/wide model)

Typical training curve:

  • Early: steep drop (learning basic statistics)
  • Middle: gradual improvement
  • Late: slow convergence

Watch for val loss flattening or diverging from train loss.

AdamW vs plain SGD

AdamW (Adam + weight decay):

  • Maintains per-parameter adaptive learning rates
  • Momentum helps escape shallow local minima
  • Weight decay provides implicit regularization

For transformers, AdamW is the standard. Plain SGD works but requires careful tuning.

optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)

Part 4: Generation

Autoregressive generation

Once trained, generate new sequences one token at a time:

@torch.no_grad()
def generate(model, idx, max_new_tokens):
    for _ in range(max_new_tokens):
        # crop context to block_size
        idx_cond = idx[:, -block_size:]
        # forward pass
        logits, _ = model(idx_cond)
        # focus on last time step
        logits = logits[:, -1, :]         # (B, vocab_size)
        probs = F.softmax(logits, dim=-1)
        # sample
        idx_next = torch.multinomial(probs, num_samples=1)
        idx = torch.cat((idx, idx_next), dim=1)
    return idx

Sampling strategies

How you pick the next token matters:

Strategy How Effect
Greedy argmax(probs) Deterministic, often repetitive
Sampling multinomial(probs) Random, more diverse
Temperature divide logits by T before softmax T < 1 → sharper / T > 1 → flatter
Top-k sample only from top k tokens Limits unlikely tokens

For nanoGPT we use plain sampling (torch.multinomial).

After training on Tiny Shakespeare for 5000 steps:

After 1 step (random model):

Part 5: Fine-Tuning

Pretrain → Fine-tune

The standard workflow for applying language models to downstream tasks:

  1. Pretrain on large unlabeled corpus (human genome) — learn general sequence statistics
  2. Fine-tune on a labeled task (promoter activity, TF binding scores) — adapt for prediction
# freeze the language model backbone
for param in model.parameters():
    param.requires_grad = False

# add a prediction head on top
model.lm_head = nn.Linear(n_embd, 1)   # regression

Fine-tuning strategies

Freeze backbone, train head

  • Fast, few parameters
  • Good if downstream task is similar to pretraining
  • Risk: limited expressivity

Full fine-tuning

  • Update all weights
  • Better for very different tasks
  • Risk: overfitting on small datasets

In Henry’s notebook: fine-tune on promoter activity and TF binding data from the Nucleotide Transformer benchmark.

Key Takeaways

  • Tokenization converts sequence to integers; vocabulary choice matters for DNA
  • Random-guess cross-entropy = \(\log(V)\) — always compute this baseline
  • The training loop is identical to unit00: forward → loss → backward → step
  • Validation loss is the true signal; watch train/val gap for overfitting
  • Autoregressive generation samples one token at a time
  • Pretrain + fine-tune is the standard genomics LM workflow
